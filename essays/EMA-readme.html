                                        # Tom Smith EMA - RSA Encryption and Signing




## Background

In public key encryption, the algorithm for encrypting and decrypting, as well as the encryption key are known but security of any message is still not compromised as there is a secret ‘private key’ (Brookshear & Brylow, 2020). The RSA (Rivest, Shamir, Adleman) algorithm is a particular implementation of this technique which relies on the complexity of factorising a number that is a product of two large primes (Anderson, 2020). 

RSA public key encryption is more secure than symmetric key encryption, but is also more resource intensive. Therefore, instead of being used for the entirety of a secure communication (such as an online card transaction over HTTPS), public key encryption will be used to communicate a symmetric key which facilitates increased efficiency.

The benefit of this system is that the prime numbers can be made increasingly large in order to stay ahead of current computer processing capabilities. Although an 829 bit number was factorised in 2020 (Boudot et al., 2020), it is commonplace for RSA encryption to use at least 2048 bits which is still considered non-factorisable (Anderson, 2020).

A drawback of RSA is that it is not forward-secure (Huang et al., 2014). Successful attacks on RSA implementations in SSL/TLS have been around for over 20 years (Klima et al., 2003) and so old secrets protected with RSA are increasingly vulnerable to losing confidentiality. Furthermore, the longer keys required for security make RSA infeasible for some applications and so elliptic curve algorithms have become preferred (Muskat, 2018).




## The Program

My program showcases two uses of the RSA algorithm, namely message encryption/decryption and signing/verification. Both of these solutions are implemented using the python RSA library created by Stüvel (2019) in order to avoid mathematical complications of implementing the RSA algorithm from scratch.

#### Encryption/Decryption

For the encryption/decryption section, the program begins by generating a new RSA public/private key pair. The program then creates a file for each of the keys and writes them to their corresponding file for later use. Since the file is designed to be used as a Jupyter notebook with code cells that can be run individually, the file writing allows the key to be stored so that the variables can be changed to another set of keys if required without losing the stored keys.

The message is then declared and encrypted by first encoding it. As described in the python RSA documentation, this step is necessary as the library operates on bytes and not strings (Stüvel, 2019). The encrypted message gets written to its own file. For the sake of integrity, the file is decrypted directly from the file so that any changes to the variable won’t impact the decryption.

#### Signature Generation

A new message is declared for the signature generation section. This message is encoded into bytes as above, hashed, and then signed with the private key (Stüvel, 2019). The hashing function recommended in documentation is SHA-1 which is now defunct (Leurant et al., 2020) and so this was replaced with SHA-256 in my implementation as described by Adeniyi et al. (2022). A one way hash is used here as the recipient will already have the message to hash themselves and just needs to verify its legitimacy. The signed message is written to its own file.


## Running Instructions

The program is a .ipynb file, and is designed to be run as a jupyter notebook so that each code cell can be run independently. 

It is also capable of running as a .py file top-to-bottom. This will require opening the file as a jupyter notebook and re-downloading the file as a .py. It may be necessary to install the additional libraries before importing them by using ‘pip’:
`pip install rsa pytest sympy`

The variables `message` and `message_unsigned` can be edited to any string.
