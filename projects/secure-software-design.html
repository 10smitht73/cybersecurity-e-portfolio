Chosen Weakness: CSRF (Cross-Site Request Forgery)

OWASP

The Open Web Application Security Project (OWASP) is a body that assesses and collates web application vulnerabilities for the purpose of spreading awareness, and improving mitigation tactics. OWASP release ‘Top Ten’ lists which document the most common categories of vulnerabilities of the time, along with prevention tips and example scenarios (OWASP, 2021). A study by Lala et al. (2021) found that implementing the security guidance of the OWASP Top Ten can change a simple web app from “highly prone to attack” to being much more resilient to attacks such as SQL injection, and cross site scripting.

CSRF (Cross-Site Request Forgery)

This post will focus on CSRF (sometimes referred to as session riding) as an attack strategy, which is the practice of exploiting a user's legitimate active session on a website to execute a malicious request (Anderson, 2020). In order to do so, the attacker needs to understand the URL pattern of the target website and create a bespoke URL containing a malicious request which is then embedded in a malicious link (Kombade & Meshram, 2012). When the user clicks on this link, their authenticated session cookie will be attached to this URL in the HTTP request and so the server will execute the request.

The flowchart below demonstrates an attack along with potential mitigations that the website may have in place:

This flow chart demonstrates that a threat actor does not need to rely on authenticating to execute an attack. The red path demonstrates a potentially successful attack which relies on multiple failures of the user/website:

  1. The user is successfully baited into clicking a link via phishing or other social engineering techniques. This outlines the need to stay vigilant and treat all links as suspicious. In corporations, it is vital to employ anti-phishing systems and conduct regular phishing training (Naqvi et al., 2023).
  2. The session token remains active when the user clicks on the link. Good practice to avoid this would be to log out of a website as soon as you are finished using it.
  3. Anti-CSRF tokens are not being used by the website. The user cannot control this, but could consider refraining from opening accounts with websites that do not employ this security feature.

Anti-CSRF Tokens

Anywhere on a website that allows a user to submit a request should have a hidden input built into the submission that contains an anti-CSRF token which is generated server-side for each request. When the user submits the request, only legitimate requests will contain the token that was generated to the user, as the attacker cannot pre-empt the randomly generated token. (Das, 2022).

Notably, enforcing HTTPS does not prevent these attacks as it only encrypts the malicious request as opposed to validating it (Microsoft, 2023). Furthermore, it may seem sensible to only permit POST requests to prevent malicious GET requests. However, the attacker can generate a malicious form (which would submit via a POST request) as opposed to a malicious link (OWASP, N.D.).

UML Models

UML models are essential to model these weaknesses within a system from multiple perspectives, and also acts as a universal language between different teams working on the same system.

Use Case Diagram: Identifies uses of the system that a threat actor may wish to exploit - points where CSRF protection should be prioritised (Brookshear & Brylow, 2020).

Class Diagram: Displays components involved in session management and request handling, which should be analysed for CSRF protection mechanisms.

Sequence Diagram: Traces the flow of actions and communications when the user interacts with the system and, therefore, when a CSRF attack could potentially occur (Brookshear & Brylow, 2020). Anti-CSRF tokens should be validated at these points (Anderson, 2020).

None of these diagrams should be exempt from a design document. However, the most important in terms of CSRF prevention is arguably the sequence diagram, as it plainly displays the exchange of requests.

Activity Notes

Read the Spears & Barki (2010) article then prepare answers to the following questions:

1. How did the authors use both Qualitative and Quantitative assessment approaches? What benefits did each approach yield?

A combination of data collection and analysis, interviews and surveys. Qualitative was necessary as there was very little known data on which to base a quantitative analysis, quantitative were then used to test the theoretical model defined from the qualitative study. Combining the two provided context and testability. Idea was that two different approaches to the same problem would yield a more whole picture.

2. What do the authors list as the advantages of involving users in the risk management process?

  • Buy-in Theory - users that participate in risk management are likely to feel more engaged in the activity in the future.
  • Organisational awareness around those who participated increased.
  • Users provided context that made the risk management more business aligned.
  • Good relationships form between users and IS professionals which can benefit the company in future IS endeavours.

More assumptions will have to be made to compensate for the lack of business context that users can provide. The advantages/risks will need to be explained thoroughly to demonstrate the need for compliance in lieu of the ‘buy-in’ that might have been.

4. Will it affect the choice of Qualitative vs. Quantitative assessment methods you utilise?

A mixture of qualitative and quantitative will be used as there is also limited historical data available in our study. However, the benefits and conclusions that can be taken from quantitative analysis cannot be ignored. A mixture of the two limits the number of assumptions that need to be made.

5. How might you mitigate any issues encountered?

Assumptions, as long as they are properly recorded, can be used to model the case around any issues. Before doing so, any issues should be investigated and researched to find if there is a previous encounter of this issue which has a known workaround.

Unit 3

Introduction to Threat Modelling and Management

Collaborative Discussion 1 Finishes!

Question: Discuss 'Industry 4.0' as explained in Kovaitė and Stankevičienė, 2019.

Summary Post

An enjoyable discussion around a topic that is very much current. Daniel, Bing and Chih have contributed to the discussion of Industry 4.0 based on the source material by Kovaitė and Stankevičienė (2019).

Daniel and Bing agreed and furthered my point that the rapid change being experienced in industry is creating obsolescence problems with both technology and staff. As Daniel explains, assessment of staff knowledge gaps is paramount to ensure successful transition into the use of new equipment and processes (Nikou et al., 2022). Although the “human factor” is now considered the main vulnerability in business information security (Jeong et al., 2019), one might argue that it is in fact the relationship between humans and technology that will become most critical - especially as cyber physical systems become more prevalent and replace manual work.

Chih presented further findings on the risks of IoT devices. It was proposed that, although businesses must carry out appropriate risk assessment before integrating any device with their network or equipment, this risk is also shared by the vendor whose company reputation could be impacted by their product being the weak link. Nurse et al. (2017) comment on approaches that can be taken in order to assess these risks successfully such as NIST SP800-30, OCTAVE, and CRAMM amongst others.

The conclusion of the discussion is in agreement with the conclusion of Kovaitė and Stankevičienė (2019). Business models must implement appropriate risk assessment when transitioning to Industry 4.0 and digitisation should be embraced with careful planning. But, of course, this is not a tick box exercise to be completed once. Just as regular patching and upgrades is key to mitigate software vulnerabilities, regular risk assessment of implemented technologies throughout their use lifetime is the most comprehensive way to understand the threats of Industry 4.0.

GDPR Case Study Activity

Case Study: Personal Data Withheld from an Access Request by Airbnb on the Basis of an Opinion Given in Confidence

What is the specific aspect of GDPR that your case study addresses?

The case study addresses the right for a data subject to access any personal data held by a data controller on them (Article 15 of GDPR) (ICO, 2016). More specifically, a data subject wants access to personal data held on them including an email of complaint written about the data subject by another third party. The data controller had to balance the right to this access with the exemptions that can apply when a confidential opinion is expressed.

How was it resolved?

As the third party had informed the data subject that they had contacted the data controller, and most of the email was considered factual, the email was ordered to be shared with the data subject as the exemption did not apply.

If this was your organisation what steps would you take as an Information Security Manager to mitigate the issue?

This reminds me of when I was working as a customer service advisor for Hargreaves Lansdown and we were trained not to write anything untoward about clients in their file even if it was an internal communication as this may have to be released to them under an access request. This should be the foundation of any company that any data on a subject should be treated as if they are constantly watching what you are doing with it. Ensuring appropriate policies should be paramount, but balancing the strictness is also important. They must be strict enough to adhere to GDPR, but if they are too strict then the Peltzman effect can take effect which makes people feel less risk averse, and in fact they may find a workaround (Stetzer & Hofmann, 1996).
